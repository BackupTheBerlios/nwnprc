Overridden Scripts

All spellscripts, special ability scripts, plus many other scripts are replaced by versions in the PRC haks. However, because of the order of priority that bioware uses (.bif -> .mod -> override -> .hak) you cannot just drop a modified version of the script into the module (it will be ignored while the hak is associated, even when opening the script). To work around this there are two alternatives

1. Place the module specific script into a hak pack with a higher priority than the PRC, a module specific hak for example. You only need the compiled script (.ncs), though it is suggested that the source script (.nss) should be in the hak as well. You can access these files by opening the module in the toolset, then looking in the modules/temp0 directory. See also Using the PRC compiler. This is the preferred option.

2. Remove the conflicting script from the PRC. This will create a “hole”, though which the script in the module will be used. However, this has to be repeated after every PRC update, and may cause issues with other modules on that server using the PRC.

For a multiplayer situation, all scripts are server side so the client does not need a modified version of the hak. It is worth noting that NWN only compares the names of hak and tlk files, not the contents.

 

Using the PRC compiler

Because of the size of many of the PRC scripts, the bioware compiler included in the toolset cannot cope. This has two major symptoms. Firstly, you get a “Too many identifiers” or “Identifier list full” error when compiling. This means that there are too many function and constant definitions to store in the right hand pane of the script editor. You can try to work around this by removing function and constant definitions, but this will lead to less readable code and is a bad idea. The second symptom is a memory error leading to the toolset becoming unresponsive. This is due to the total length of a script after includes have been added but before comments have been removed. Again, you can try to work around it by removing unnecessary includes and by removing comments, but this will lead to less readable and maintainable code; this is a bad idea.

The PRC compiler (nwnnsscomp.exe available at http://nwvault.ign.com/Files/other/data/1098057351000.shtml) is very similar to the bioware command line compiler included in NWN (look for clcompile.exe in the utils directory). The PRC compiler does have the major advantage that you can specify an include directory. Both compilers are designed to work as dos utilities; I will not cover basic dos commands and how to access the command prompt.

Nwnnsscomp has the following command parameters:

Usage:
nwnnsscomp [-cdegoqx] [-t#] [-v#] [[-i pathspec] ...] infile [outfile]

pathspec - semicolon separated list of directories to search for files.
infile - name of the input file.
outfile - name of the output file.
-c - Compile the script (default)
-d - Decompile the script (can't be used with -c)
-e - Enable non-Bioware extensions
-g - Don't produce ndb debug file
-i - Add Search Path
-o - Optimize the compiled source
-q - Silence most messages
-x - Extract script from NWN data files
-vx.xx - Set the version of the compiler
-t1 - Perform a compilation test with BIF scripts
-t2 - Perform a compilation test with the given module
-t3 - Optimization space saving report with the given module
-t4 - Perform a compilation test with the given file or files

I would suggest that the easiest way to work with it is as follows:

Copy nwnnsscomp.exe to your modules directory.

Create a subdirectory in modules named prcinc and extract the contents of prc_include.hak to it.

Create a batch file (a text file with the .bat extension) in the module directory.

Enter the following into the batch file:

copy nwnnsscomp.exe temp0\nwnnsscomp.exe
cd temp0
nwnnsscomp -g -i ..\prcinc *.nss
pause
del nwnnsscomp.exe
cd..

Disable the toolset compiler (go to tools -> options -> script editor and deselect “Automatically Compile Scripts on Save”. Then go to general and make sure “Build Module on Save” is disabled as well).

From now on, use the batch file to compile your scripts, rather than the toolset. Not only can it cope with larger scripts, it also compiles faster. For speedier access to the batch file, place a shortcut on your desktop.

 

Using the new races

Unfortunately, bioware have hardcoded the classes and races you can select at level 1. For single player and localvault games this can be easily bypassed by using the java Character Creator (available at http://nwvault.ign.com/Files/other/data/1050041596353.shtml) to create the character outside of the game. For servervault games however, this is not suitable. The players would have to create their character locally, then email the .bic file to the server admin, who would have to check the file to prevent cheats/exploits, and then place it into the servervault. Since the character creator is java, it my be possible to integrate it into a web page, but this is not being attempted at the moment (any volunteers?). The solution is an in-game conversation driven character creator (available at http://nwvault.ign.com/Files/scripts/data/1097822211000.shtml). Neither of these options are going to be documented here, please refer to the documentation included in each respective package and the PRC forums.

With these new races, there are several things to note.

The stat changes are true stat changes, the +12 cap is on top of these.

Scripted race commands (such as turn undead), will use the base race (Undead, outsider, humanoid). Since some races can belong to more than one category (Teiflings are Humanoid (human) and Outsiders for example), there is a priority pyramid, shown below. Groups on the same level are normally nutually exclusive, but not always (Undead and Construct for example. A Construct is never alive, so it cannot be brought back to life. However, since you can create a Construct out of dead beings, a Bone Golem for example, so it is possible to be in both groups in rare exceptions). If a creature belongs to groups on two levels, the higher level has priority. However, it is still a member of the lower group if it is not a direct competition. For example, a Tannaruk is both an Outsider and a Humanoid (orc). Thus it is affected by planar turning and planar binding, but it is also eligible for Orc–specific prestige classes, such as Orc Warlord and Eye of Gruumsh.

The subrace field on a character has no effect on what race it is. This is the same as the default bioware races.

Those races that are traditionally considered subraces (Drow as a subrace of elves for example), are implemented as true separate races. This is a bioware limitation.

Hardcoded race settings, such as +X vs Y itemproperties (+2 attack vs elves) and ranger hated enemy, do not effect any of the new races. This is not a significant problem in most cases since there is little Player vs Player and few NPCs use the PRC races. Again, this is a bioware limitation.

Many of the races are far more powerful than the standard bioware races. In Pen’n’Paper, this is accounted for the Level Adjustment and Effective Character Level. LA is the added to the number of hit dice a character has to determine that characters ECL. The ECL is then used to determine XP required to level and for comparing characters within a party. For example, a human level 8 barbarian is considered equal to a pixie (LA +4) level 4 rogue.
